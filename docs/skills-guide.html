安装

npx skills add szj2ys/skills

这会把所有 skills 下载到你当前项目的 .agents/skills/ 目录。之后在 Cursor、Codex 或其他支持 skills 的 AI 编码工具里就能直接调用。

如果你只想用部分分类，可以只安装特定子目录（取决于你的工具是否支持）。

怎么用这份指南

这份文档解决三个问题：

  1. 场景识别 — 我现在要做的事属于哪种场景？
  2. Skill 选择 — 这个场景该用哪些 skill？
  3. 执行顺序 — 这些 skill 按什么顺序串起来？
提示：不是每个任务都需要走完所有步骤。简单 bug 只需要 diagnose + tdd + commit；一个全新功能才需要走完整条线。按需取用，不要教条。
元 skill：using-agent-skills（在 productivity/ 里）是一棵完整的工程决策树，能帮你自动判断当前任务该用哪个 skill。它是这份指南的机器可读版本。

第一步：判断你该走哪条路

拿到一个任务，先回答左边的问题，沿着箭头走就能找到对应的场景。

任务到来

不知道要做什么？ → grill-me / interview-me → idea-refine → spec-driven-development
要做一个新功能？ → 场景 A
东西坏了？ → 场景 B
代码太乱想重构？ → 场景 C
要搞 SEO / 内容 / 增长？ → 场景 D
要做 UI / 设计 / 改版？ → 场景 E
要写 PRD / 排优先级 / 做规划？ → 场景 F
要保存上下文 / 交接 / 复盘？ → 场景 G
用特定语言/框架写代码？ → 场景 H
要生成 PDF / 文档 / slides？ → 场景 I

场景 A：做新功能（从需求到上线）

触发词：「做个 XX 功能」「加个 XX」「实现 XX」

完整流程

定义spec-driven-development → 规划planning-and-task-breakdown → 构建context-engineering → 构建incremental-implementation
构建tdd → 审查code-review-and-quality → 审查code-simplify → 发布commit
发布pull-requests / ship → 发布deploy-to-vercel / land-and-deploy → 发布canary

各步骤说明

步骤用什么触发词可以跳过吗
定义spec-driven-development需求模糊、改动大、涉及架构决策小改动可以跳过
定义grill-with-docs想验证方案是否合理大多数情况可以跳过
规划planning-and-task-breakdown任务 > 3 个文件，或需要并行单文件改动可以跳过
构建context-engineeringagent 输出质量下降、换模块上下文清晰时可跳过
构建incremental-implementation任何多文件改动核心步骤，不建议跳
构建source-driven-development用框架 API、不确定最佳实践熟悉的 API 可跳过
构建doubt-driven-development高风险决策、安全相关低风险可跳过
构建tdd写功能代码时同步写测试核心步骤，不建议跳
审查code-review-and-quality合并前必做五维审查核心步骤
审查code-simplify代码太复杂需要简化代码已简洁可跳过
发布ship一键：测试 → 审查 → bump → commit → PR不想全自动就用 commit + PR 分步走
发布canary部署后监控非生产环境可跳过

快速版（小功能）

incremental-implementation → tdd → commit

场景 B：修 Bug / 定位问题

触发词：「这里有个 bug」「XX 不工作了」「报错了」「性能下降了」

流程

定位diagnose → 定位investigate → 修复tdd（写回归测试） → 审查code-review-and-quality → 发布commit → pull-requests
步骤用什么触发词
定位diagnose硬 bug、不确定根因 — 先建反馈循环，再假设、验证、修复
定位investigate系统性调查，比 diagnose 轻量
定位zoom-out看不懂这段代码 — 先看模块全景
修复tdd先写一个会失败的回归测试，再修
审查code-review-and-quality修完后审查修复本身 + 回归测试
关键：diagnose 的核心是 Phase 1 — 建反馈循环。如果你能写一个失败测试来复现 bug，这个 bug 就 90% 修完了。

pstack 原则（19 条工程信条）

来自 cursor/plugins 仓库的 pstack 原则。这些不是流程 skill，而是你做工程决策时的指导原则。遇到「该不该这样做」的纠结时，回来翻翻。

原则核心含义什么时候用
principle-experience-first先定义用户体验，再写代码开始任何新功能前
principle-subtract-before-you-add先删后加 — 代码越多越难维护准备加新代码时
principle-fix-root-causes修根因，不修症状修 bug 时
principle-prove-it-works不要说「应该能跑」— 要有证据（测试、截图、日志）声称「完成了」之前
principle-foundational-thinking先把地基打好 — 依赖、类型、接口先稳定规划实现顺序时
principle-exhaust-the-design-space先把设计空间探完再选方案 — 不要想到第一个方案就开干架构决策时
principle-encode-lessons-in-structure把教训编码进代码结构（类型、测试、接口约束），而不是文档里踩坑之后
principle-guard-the-context-window保护 agent 的上下文窗口 — 控制文件大小、减少噪音agent 输出质量下降时
principle-minimize-reader-load减少阅读者的认知负荷 — 命名清晰、结构简单写代码和文档时
principle-make-operations-idempotent操作要幂等 — 重复执行不会出错写脚本、API、部署流程时
principle-never-block-on-the-human不要等人 — 能推就推，需要人决策时明确标注agent 遇到不确定时
principle-boundary-discipline严格的边界纪律 — 每个模块只管自己的事设计模块接口时
principle-laziness-protocol能懒就懒 — 不做不必要的事准备「顺手」做点什么之前
principle-outcome-oriented-execution以结果为导向 — 不要为了写代码而写代码评估任务价值时
principle-redesign-from-first-principles从第一性原理重新设计 — 不要被现有实现锚定重构陷入困境时
principle-build-the-lever造杠杆工具 — 一次投入，多次受益发现自己在重复做同一件事时
principle-separate-before-serializing-shared-state共享状态先分离再序列化处理共享状态持久化时
principle-type-system-discipline类型系统纪律 — 用类型约束代替运行时检查设计数据结构时
principle-migrate-callers-then-delete-legacy-apis先迁移调用方，再删旧 API — 不要一步到位废弃旧接口时
