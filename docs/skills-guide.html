安装

npx skills add szj2ys/skills

这会把所有 skills 下载到你当前项目的 .agents/skills/ 目录。之后在 Cursor、Codex 或其他支持 skills 的 AI 编码工具里就能直接调用。

如果你只想用部分分类，可以只安装特定子目录（取决于你的工具是否支持）。

怎么用这份指南

这份文档解决三个问题：

  1. 场景识别 — 我现在要做的事属于哪种场景？
  2. Skill 选择 — 这个场景该用哪些 skill？
  3. 执行顺序 — 这些 skill 按什么顺序串起来？
提示：不是每个任务都需要走完所有步骤。简单 bug 只需要 diagnose + tdd + commit；一个全新功能才需要走完整条线。按需取用，不要教条。
元 skill：using-agent-skills（在 agent/ 里）是一棵完整的工程决策树，能帮你自动判断当前任务该用哪个 skill。它是这份指南的机器可读版本。

怎么调用 skill

使用 $skill-name 语法显式触发 skill。可以在对话中直接写，也可以在 prompt 文件顶部写。以下是三种典型的调用方式。

调用方式 1：单个 skill

最简单的方式 — 在对话中用 $skill-name 触发一个 skill：

$diagnose 我的登录页面在高并发时报 500 错误，本地无法复现

$tdd 为用户权限检查函数写测试

$spec-driven-development "给系统加一个多租户功能"

$seo-audit https://myapp.com

$grill-with-docs 我打算用事件溯源做订单系统

$frontend-design 设计一个定价页面，参考 Linear 风格

$code-review-and-quality

我来拆解并开始实现 auth/ 的改造

$canary 监控灰度发布

调用方式 2：组合调用（多 skill 串联）

一个复杂任务往往需要多个 skill 按顺序执行。每个 skill 完成后，你可以手动触发下一个，也可以让 Agent 推荐下一步：

# SEO 全流程
$seo-audit https://myapp.com
$keyword-research "AI project management tools"
$site-architecture

另一个例子 — 做一个新功能从头到尾：

# 定义需求
$spec-driven-development "给用户加一个角色权限系统"

# 规划任务
$planning-and-task-breakdown 实现角色权限系统

# 测试驱动开发
$tdd 为权限检查函数写测试

# 审查 + 提交
$code-review-and-quality
$commit

调用方式 3：组合调用（同一 prompt 里同时触发多个）

当你需要 Agent 同时做多件事时，可以把多个 skill 名放在同一个 prompt 里：

# 同时做 SEO 审计和关键词研究
$seo-audit + $keyword-research "AI project management tools" https://myapp.com

# 设计 + 生成图片
$frontend-design + $image-generation "pricing page hero"

# 审计 + 简化
$code-audit + $code-simplify src/auth/

# 关键词研究 + 内容策略 + AI SEO
$keyword-research + $content-strategy + $ai-seo "project management tools"

# 产品探索 + PRD
$grill-me + $idea-refine + $to-prd
注意：$skill-name 是显式触发方式。即使不写 $，只要你的 prompt 里包含 skill 的触发词（比如「诊断这个 bug」「写个测试」「做 SEO 审计」），Agent 也会自动匹配对应的 skill。但显式触发更可靠，尤其是在需要精确控制执行顺序时。下面各场景表格中的「调用示例」列都是可以直接复制使用的。

pstack 原则（19 条工程信条）

来自 agent/ 目录的 pstack 原则。这些不是流程 skill，而是你做工程决策时的指导原则。遇到「该不该这样做」的纠结时，回来翻翻。

原则核心含义什么时候用
principle-experience-first先定义用户体验，再写代码开始任何新功能前
principle-subtract-before-you-add先删后加 — 代码越多越难维护准备加新代码时
principle-fix-root-causes修根因，不修症状修 bug 时
principle-prove-it-works不要说「应该能跑」— 要有证据（测试、截图、日志）声称「完成了」之前
principle-foundational-thinking先把地基打好 — 依赖、类型、接口先稳定规划实现顺序时
principle-exhaust-the-design-space先把设计空间探完再选方案 — 不要想到第一个方案就开干架构决策时
principle-encode-lessons-in-structure把教训编码进代码结构（类型、测试、接口约束），而不是文档里踩坑之后
principle-guard-the-context-window保护 agent 的上下文窗口 — 控制文件大小、减少噪音agent 输出质量下降时
principle-minimize-reader-load减少阅读者的认知负荷 — 命名清晰、结构简单写代码和文档时
principle-make-operations-idempotent操作要幂等 — 重复执行不会出错写脚本、API、部署流程时
principle-never-block-on-the-human不要等人 — 能推就推，需要人决策时明确标注agent 遇到不确定时
principle-boundary-discipline严格的边界纪律 — 每个模块只管自己的事设计模块接口时
principle-laziness-protocol能懒就懒 — 不做不必要的事准备「顺手」做点什么之前
principle-outcome-oriented-execution以结果为导向 — 不要为了写代码而写代码评估任务价值时
principle-redesign-from-first-principles从第一性原理重新设计 — 不要被现有实现锚定重构陷入困境时
principle-build-the-lever造杠杆工具 — 一次投入，多次受益发现自己在重复做同一件事时
principle-separate-before-serializing-shared-state共享状态先分离再序列化处理共享状态持久化时
principle-type-system-discipline类型系统纪律 — 用类型约束代替运行时检查设计数据结构时
principle-migrate-callers-then-delete-legacy-apis先迁移调用方，再删旧 API — 不要一步到位废弃旧接口时
